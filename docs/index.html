Adopt In 5 Minutes

Pick the path that matches how you ship.

The consumer path should not look like contributor setup. Use the CLI if you want a local or CI script immediately. Use the composite Action if you want a copy-paste GitHub workflow in an existing repo.

Use it from your existing repo

Install the package, download Playwright once, and keep the generated artifacts in your CI output.

npm i -D web-quality-gatekeeper
npx playwright install
npx wqg audit https://your-site.example \
  --out artifacts \
  --config configs/default.json

Use the GitHub Action path

Start from the composite Action when you want a repo-native check with artifact upload and policy controls.

jobs:
  web-quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2
        with:
          persist-credentials: false
      - uses: Jahrome907/web-quality-gatekeeper@v3
        with:
          url: https://your-site.example
          baseline-dir: baselines

Action consumers keep the same protected output names and can still inspect summary.json, summary.v2.json, and report.html.

Workflow

One run produces both human review surface and automation surface.

The tool is intentionally opinionated about what gets emitted so a maintainer, a CI workflow, and a downstream dashboard are all looking at the same run.

1

Target + policy

Resolve CLI input, built-in policy overlays, and SSRF-safe target rules before work begins.

2

Capture evidence

Run Playwright screenshots, axe scanning, Lighthouse budgets, and optional visual diff checks.

3

Write stable artifacts

Emit report.html, summary.json, summary.v2.json, and supporting evidence files.

4

Gate release decisions

Use the same outputs locally, in Actions, and in case-study or contract-validation workflows.

Why Trust It

Trust comes from explicit contracts, reproducibility, and boring guardrails.

This repo is trying to read like disciplined engineering work, not a feature dump. The public story should point directly at the rules, tests, and compatibility documents that make the outputs credible.

Why this exists

Many QA wrappers stop at pass/fail. That leaves reviewers to chase screenshots, logs, and schema assumptions across tools. WQG keeps the evidence bundle in one place so decisions are inspectable and repeatable.

  • One run for smoke, a11y, perf, and visual review.
  • Artifact-first design so humans and automation consume the same truth.
  • Security-aware behavior for internal targets, auth, and workflow publication.

Differentiators that are easy to miss in a feature grid

  • Schema and docs are contract-checked, not hand-waved.
  • Workflow behavior is hardened for forks, token limits, and release-tag safety.
  • CLI, Action, and summary contracts are explicitly held stable in this cycle.
  • Case-study claims are backed by reproducibility docs and checked-in sample evidence.

Compatibility promises

The current release track keeps these public surfaces stable by default:

  • wqg audit remains the primary CLI entrypoint.
  • summary.json stays backward-compatible.
  • summary.v2.json evolves additively.
  • action.yml inputs and outputs remain compatible unless explicitly extended.
